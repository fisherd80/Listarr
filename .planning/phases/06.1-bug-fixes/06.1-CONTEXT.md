<decisions>
## Implementation Decisions

### Tag Behavior
- Override tag **replaces** default service tag (not merges)
- Lists with no override tag **fall back** to service default tag
- Keep service-level default tag in Config page as fallback
- Tag normalization applies always (lowercase, hyphens): "My Tag" → "my-tag"
- Auto-create missing tags in Radarr/Sonarr (existing Phase 3.1 behavior)

**Bug discovered:** Tag changes not saving on edit form — investigate as part of fix #1

### Logging Configuration
- Default log level: **INFO** when LOG_LEVEL not set
- Console output only (Docker best practice, no file logging)
- Human-readable format: timestamp + level + message

**Behavior by level:**
- **INFO (normal):** Job start/complete/fail events only — no HTTP request logs
- **WARNING:** Warnings and errors only
- **DEBUG:** Full verbose logging including HTTP requests, individual item imports

**Key issue:** HTTP request logs appearing at INFO level when they should be DEBUG-only

### UI Feedback Issues (Monitoring)
- Bugs #4, #5, #6 (run status, toast notifications, edit navigation) may have been resolved during Wave 2 work
- Monitor during Phase 6.1 execution
- If issues reappear, escalate to active fixes

### Claude's Discretion
- How to handle blank/empty override tag field (interpret as "use default" or "explicitly no tag")
- Which loggers get LOG_LEVEL applied (all vs just app loggers)
- Technical approach to suppressing library logs at non-DEBUG levels

</decisions>
