## Bug Analysis

### Bug #1: Override Tag Duplicating

**Location:** `listarr/services/import_service.py`, lines 114-119

**Root Cause:** The `resolve_import_settings()` function adds BOTH the default tag AND the override tag to the tags list:
```python
# Build tags list from default and override
tags = []
if import_settings and import_settings.default_tag_id:
    tags.append(import_settings.default_tag_id)  # Adds default
if list_obj.override_tag_id and list_obj.override_tag_id not in tags:
    tags.append(list_obj.override_tag_id)  # Adds override (but default is already there)
```

**Correct Behavior (per CONTEXT.md):**
- Override tag REPLACES default service tag (not merges)
- Lists with no override tag fall back to service default tag

**Fix Pattern:**
```python
# Resolve tags - override REPLACES default, not merges
tags = []
if list_obj.override_tag_id:
    # Use override tag only
    tags.append(list_obj.override_tag_id)
elif import_settings and import_settings.default_tag_id:
    # Fall back to service default tag
    tags.append(import_settings.default_tag_id)
```

**Related Issue:** Tag changes not saving on edit form - investigate in `lists_routes.py` wizard_submit handler.

**Confidence:** HIGH - Code location and fix verified against codebase

### Bug #3: Logging Configuration

**Location:** `listarr/__init__.py`, lines 39-43

**Current State:**
```python
# Configure logging
logging.basicConfig(
    level=logging.INFO,  # Hard-coded, ignores LOG_LEVEL env var
    format='%(asctime)s %(name)s %(levelname)s: %(message)s'
)
```

**Root Cause:** LOG_LEVEL environment variable is not read; level is hardcoded to INFO.

**HTTP Request Logs Issue:** pyarr library (used for Radarr/Sonarr API calls) logs HTTP requests at DEBUG level via httpx/httpcore loggers. When root logger is DEBUG, all library logs appear.

**Fix Pattern (per CONTEXT.md decisions):**
```python
import os
import logging

def configure_logging():
    """Configure logging based on LOG_LEVEL environment variable."""
    # Default to INFO, respect LOG_LEVEL env var
    log_level_str = os.environ.get('LOG_LEVEL', 'INFO').upper()
    log_level = getattr(logging, log_level_str, logging.INFO)

    # Configure root logger
    logging.basicConfig(
        level=log_level,
        format='%(asctime)s %(name)s %(levelname)s: %(message)s'
    )

    # Suppress noisy library logs unless DEBUG mode
    if log_level > logging.DEBUG:
        # Suppress HTTP request logs from libraries
        logging.getLogger('httpx').setLevel(logging.WARNING)
        logging.getLogger('httpcore').setLevel(logging.WARNING)
        logging.getLogger('urllib3').setLevel(logging.WARNING)
        logging.getLogger('werkzeug').setLevel(logging.WARNING)
```

**Behavior by Level (per CONTEXT.md):**
- **INFO (default):** Job start/complete/fail events only, no HTTP request logs
- **WARNING:** Warnings and errors only
- **DEBUG:** Full verbose logging including HTTP requests

**Confidence:** HIGH - Flask logging patterns verified via official docs

## Architecture Patterns

### Pattern 1: Environment Variable Configuration

**What:** Read configuration from environment variables with sensible defaults

**When to use:** Application-wide settings that vary by deployment environment

**Example:**
```python
# Source: Python stdlib / Flask patterns
import os

# Read with default
log_level = os.environ.get('LOG_LEVEL', 'INFO').upper()

# Convert to logging constant
level = getattr(logging, log_level, logging.INFO)
```

### Pattern 2: Hierarchical Logger Suppression

**What:** Set log levels on specific named loggers to suppress noisy libraries

**When to use:** When libraries log too verbosely at INFO/DEBUG level

**Example:**
```python
# Source: Python logging documentation
import logging

# Only suppress if not in DEBUG mode
if effective_level > logging.DEBUG:
    for logger_name in ['httpx', 'httpcore', 'werkzeug']:
        logging.getLogger(logger_name).setLevel(logging.WARNING)
```

### Pattern 3: Conditional Tag Assignment

**What:** Use override value if set, otherwise fall back to default

**When to use:** Any setting that has per-item override and service-wide default

**Example:**
```python
# Override REPLACES default, not merges
if item.override_value:
    result = item.override_value
elif defaults and defaults.default_value:
    result = defaults.default_value
else:
    result = None
```

## Common Pitfalls

### Pitfall 1: Modifying Root Logger After Import

**What goes wrong:** Libraries import logging before your configuration runs, using default settings.

**Why it happens:** Python logging is configured at import time for some libraries.

**How to avoid:** Configure logging early in app factory (before importing other modules).

**Warning signs:** Library logs appearing despite configuration.

### Pitfall 2: Forgetting Empty String Check for Override

**What goes wrong:** Empty string `""` evaluates truthy in some contexts, treated as valid override.

**Why it happens:** Form fields submit empty strings, not None.

**How to avoid:** Check both `if value and value.strip()` or use `or None` pattern.

**Warning signs:** Blank override treated as intentional empty tag.

### Pitfall 3: LocalStorage/Server State Desync

**What goes wrong:** Client thinks job is running but server says idle (or vice versa).

**Why it happens:** Page reload, browser close, server restart during job.

**How to avoid:** Server is source of truth; client localStorage is hint only.

**Warning signs:** Buttons stuck in "Running" state, multiple polls for same job.

## Claude's Discretion Recommendations

Per CONTEXT.md, these decisions are at Claude's discretion:

### 1. Blank/Empty Override Tag Field Handling

**Recommendation:** Treat blank/empty as "use default"

**Rationale:**
- Users expect empty field = default behavior
- Explicitly "no tag" is edge case; could add explicit checkbox if needed later
- Current wizard already uses `null` for "use default" pattern

### 2. Which Loggers Get LOG_LEVEL Applied

**Recommendation:** Apply to app loggers; suppress libraries unless DEBUG

**Rationale:**
- Library logs (httpx, werkzeug) are noise at INFO level
- Users want to see their job logs, not HTTP request details
- DEBUG mode = full transparency for troubleshooting

### 3. Technical Approach to Suppressing Library Logs

**Recommendation:** Set individual logger levels, not filters

**Rationale:**
- Simpler than custom Filter classes
- Standard Python logging pattern
- Easy to maintain and understand
