**docker-compose.yml — Full rewrite:**

Replace the current minimal docker-compose.yml with a production-ready version. Use this structure:

```yaml
services:
  listarr:
    build: .
    container_name: listarr
    ports:
      - "5000:5000"
    volumes:
      - ./instance:/app/instance
    environment:
      - FLASK_DEBUG=false
      - LOG_LEVEL=INFO
      # - TZ=America/New_York          # Timezone for cron schedules (default: UTC)
      # - SECURE_COOKIES=true           # Enable when behind HTTPS reverse proxy
      # - FERNET_KEY=                   # Override encryption key (auto-generated if missing)
      # - LISTARR_SECRET_KEY=           # Override Flask secret key (auto-generated if missing)
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "python", "-c", "import requests; r=requests.get('http://localhost:5000/health', timeout=5); r.raise_for_status()"]
      interval: 30s
      timeout: 10s
      start_period: 30s
      retries: 3
```

Key decisions:
- Keep bind mount `./instance:/app/instance` (not named volume) — homelab users want easy access to DB file for backups
- Keep `restart: unless-stopped` (already present)
- Add explicit `container_name: listarr` for predictable container naming
- Document all env vars with comments explaining purpose and defaults
- Healthcheck uses same Python requests pattern as Dockerfile but with explicit compose stanza for visibility

**Dockerfile — Update HEALTHCHECK start_period:**

Find the existing `HEALTHCHECK` instruction and update `start-period` from `5s` to `30s`. This gives APScheduler and database initialization time to complete on cold starts. Keep all other HEALTHCHECK parameters (interval, timeout, retries) as-is unless they're less than: interval=30s, timeout=10s, retries=3.

Do NOT change anything else in the Dockerfile — the non-root user, multi-stage build, entrypoint, and gosu usage are all correct.

**Fresh volume boot path verification:**

Read through `entrypoint.sh` and `setup.py` to verify the boot flow:
1. Empty volume → entrypoint.sh runs setup.py → generates .fernet_key and .secret_key
2. create_app() → db.create_all() creates tables
3. before_app_request check_setup() → redirects to /setup if no users
4. User creates account → auto-login → redirect to dashboard

If any gap is found in this flow, document it in the SUMMARY but do NOT change entrypoint.sh or setup.py unless there's a clear bug.
