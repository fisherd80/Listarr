# Phase 03.1 Plan 01: Tag Storage Implementation Summary

**One-liner:** Tag text inputs on Config page with create-if-missing pattern using normalized labels (lowercase, hyphens)

## What Was Built

Implemented full tag storage functionality on the Config page for both Radarr and Sonarr services:

1. **Service Layer Functions:**
   - Added `create_or_get_tag_id()` to both radarr_service.py and sonarr_service.py
   - Normalizes tag labels: lowercase, spaces to hyphens, strip consecutive hyphens
   - Implements create-if-missing pattern: searches existing tags, creates if not found
   - Returns tag ID on success, None on error

2. **Backend Routes:**
   - Updated `fetch_radarr_import_settings()` and `fetch_sonarr_import_settings()` to fetch and return tag labels
   - Updated `save_radarr_import_settings()` and `save_sonarr_import_settings()` to handle tag creation/lookup
   - Added error handling for tag operations with user-friendly messages

3. **Frontend:**
   - Enabled tag text inputs (removed `disabled` attribute)
   - Added helper text explaining tag creation and allowed characters
   - Updated JavaScript to load tag labels from saved settings
   - Updated save handlers to include tag_label in POST requests
   - Sends `null` for empty tags to allow clearing

## Technical Implementation

### Tag Normalization Logic

```python
# 1. Convert to lowercase
# 2. Replace spaces with hyphens
# 3. Remove consecutive hyphens
# 4. Strip leading/trailing hyphens
normalized_label = tag_label.lower().replace(" ", "-")
while "--" in normalized_label:
    normalized_label = normalized_label.replace("--", "-")
normalized_label = normalized_label.strip("-")
```

### Create-If-Missing Pattern

1. Fetch all existing tags from Radarr/Sonarr
2. Search for tag with case-insensitive label match
3. If found: return existing tag ID
4. If not found: create new tag via API
5. Return new tag ID

### Data Flow

```
User enters tag → Frontend validates → Backend normalizes →
API call to Radarr/Sonarr → Tag created/found →
ID stored in MediaImportSettings.default_tag_id
```

## Decisions Made

### Tag Normalization Strategy

**Decision:** Convert all tags to lowercase with hyphens, removing spaces and consecutive hyphens.

**Rationale:** Radarr and Sonarr APIs reject tags with spaces or uppercase letters with 400 errors. This normalization ensures all tags are valid before API submission.

**Impact:** Users can enter tags naturally (e.g., "My Tag") and system converts to valid format (e.g., "my-tag").

### Create-If-Missing vs. Pre-existing Tags

**Decision:** Automatically create tags if they don't exist in Radarr/Sonarr.

**Rationale:**
- Better user experience - no need to pre-create tags manually
- Consistent with import workflow expectations
- Prevents failed imports due to missing tags

**Alternative Considered:** Require users to create tags in Radarr/Sonarr first, then select from dropdown.

**Why Rejected:** Extra steps, poor UX, breaks workflow continuity.

## Integration Points

### Database Schema
- Uses existing `MediaImportSettings.default_tag_id` column (foreign key to tags table)
- Tag ID stored as integer, NULL allowed for no tag

### APIs Used
- `RadarrAPI.get_tag()` - fetch all tags
- `RadarrAPI.create_tag(label)` - create new tag
- `SonarrAPI.get_tag()` - fetch all tags
- `SonarrAPI.create_tag(label)` - create new tag

### Frontend-Backend Contract
- GET `/config/{service}/import-settings` returns `tag_label` string
- POST `/config/{service}/import-settings` accepts `tag_label` string or null
- Null tag_label clears the tag (sets default_tag_id to NULL)

## Lessons Learned

1. **API Validation Matters:** Radarr/Sonarr APIs are strict about tag format. Normalization at service layer prevents frontend errors.

2. **Create-If-Missing Pattern Works Well:** Users don't need to understand Radarr/Sonarr tag management - system handles it automatically.

3. **Helper Text is Essential:** Users need to know what characters are allowed and that tags auto-create.
