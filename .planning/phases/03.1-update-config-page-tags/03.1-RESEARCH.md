# Phase 3.1: Update Config Page Tags - Research

**Researched:** 2026-01-23
**Domain:** Radarr/Sonarr tag management with create-if-missing pattern
**Confidence:** HIGH

## Summary

This phase adds tag storage to the Config page, which was missed during the original config page implementation. The core challenge is implementing a "create-if-missing" pattern where users enter tag names (not select from dropdowns), and the system checks if the tag exists in Radarr/Sonarr, creating it if necessary before storing the tag ID in the database.

Research confirms that:
- **PyArr library** provides complete tag CRUD operations via `get_tag()` and `create_tag()` methods
- **Radarr/Sonarr APIs** enforce strict tag validation rules (lowercase a-z, 0-9, hyphens only)
- **Tag validation changed recently** (2025-2026) to reject spaces and uppercase letters with 400 errors
- **Create-if-missing pattern** requires case-insensitive label matching due to API normalization
- **Database model** already supports tag storage (`MediaImportSettings.default_tag_id`)

The existing codebase already uses PyArr for quality profiles and root folders, making tag integration straightforward. The wizard already fetches tags as dropdowns, so the Config page will follow the same service-layer pattern.

**Primary recommendation:** Use PyArr's `get_tag()` to fetch all tags, search for case-insensitive label match, and use `create_tag(label)` if not found. Store the resulting tag ID in `MediaImportSettings.default_tag_id`.

### Pattern 1: Create-If-Missing Tag Pattern
**What:** Check if tag exists by label (case-insensitive), create if missing, return tag ID
**When to use:** Any time user enters a tag name that needs to be stored as an ID

**Implementation:**
```python
# Source: PyArr base.py (https://github.com/totaldebug/pyarr/blob/main/pyarr/base.py)
# Pattern: Existing codebase services/radarr_service.py and sonarr_service.py

def create_or_get_tag_id(base_url: str, api_key: str, tag_label: str):
    """
    Creates tag if it doesn't exist, or returns existing tag ID.
    Performs case-insensitive label matching.

    Args:
        base_url (str): Base URL of Radarr/Sonarr
        api_key (str): API key
        tag_label (str): Tag label to create or find

    Returns:
        int: Tag ID, or None on error
    """
    # Ensure base_url ends with slash
    if not base_url.endswith("/"):
        base_url += "/"

    try:
        # Initialize API client (RadarrAPI or SonarrAPI)
        api = RadarrAPI(host_url=base_url, api_key=api_key)

        # Normalize label to match API requirements (lowercase, no spaces)
        normalized_label = tag_label.lower().strip().replace(" ", "-")

        # Get all existing tags
        tags = api.get_tag()

        # Search for existing tag (case-insensitive)
        for tag in tags:
            if tag["label"].lower() == normalized_label.lower():
                return tag["id"]

        # Tag doesn't exist - create it
        new_tag = api.create_tag(label=normalized_label)
        return new_tag["id"]

    except Exception as e:
        logger.error(f"Error creating/getting tag '{tag_label}': {e}", exc_info=True)
        return None
```

### Anti-Patterns to Avoid
- **Don't use dropdown for tags:** Requirements specify text input for create-if-missing functionality
- **Don't skip label normalization:** Radarr/Sonarr enforce lowercase with hyphens; normalize before API calls
- **Don't assume case-sensitive matching:** Tags are normalized to lowercase, so matching must be case-insensitive
- **Don't create duplicate tags:** Always search existing tags before creating new ones

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Radarr/Sonarr API calls | Custom HTTP client with requests | PyArr library (RadarrAPI/SonarrAPI) | PyArr handles API versioning, authentication, error responses, and provides typed methods |
| Tag label validation | Custom regex validation | Normalize to API requirements (lowercase, replace spaces with hyphens) | API enforces validation; better to normalize than reject user input |
| Tag duplicate detection | Simple string equality | Case-insensitive label matching | Tags are normalized to lowercase by API, so "MyTag" and "mytag" are the same |
| Form CSRF protection | Manual token generation | Flask-WTF built-in CSRF | Already used throughout codebase, provides automatic token generation |

**Key insight:** PyArr abstracts all API complexity. The library is already used for quality profiles and root folders, so tag operations follow the same pattern. Don't reinvent API interactions.

## Common Pitfalls

### Pitfall 1: Tag Validation Errors (400 Bad Request)
**What goes wrong:** Creating tags with spaces, uppercase letters, or special characters fails with 400 error
**Why it happens:** Radarr/Sonarr recently (2025-2026) started enforcing strict validation: only lowercase a-z, 0-9, and hyphens allowed
**How to avoid:** Normalize tag labels before API calls - convert to lowercase and replace spaces with hyphens
**Warning signs:** Error message "Allowed characters a-z, 0-9 and -" or "RegularExpressionValidator"

**Example:**
```python
# BAD - will fail with 400 error
tag_label = "My Tag"
api.create_tag(label=tag_label)  # Spaces not allowed

# GOOD - normalize first
tag_label = "My Tag".lower().replace(" ", "-")  # "my-tag"
api.create_tag(label=tag_label)  # Success
```

### Pitfall 2: Creating Duplicate Tags
**What goes wrong:** User enters tag name that already exists but with different case (e.g., "MyTag" vs "mytag"), creating duplicates
**Why it happens:** Not performing case-insensitive search before creating tag
**How to avoid:** Always fetch existing tags and compare labels case-insensitively before creating
**Warning signs:** Multiple tags with similar names appearing in Radarr/Sonarr UI

**Example:**
```python
# BAD - creates duplicate if "mytag" exists
new_tag = api.create_tag(label="MyTag")

# GOOD - check first
existing_tags = api.get_tag()
normalized_label = "MyTag".lower()
for tag in existing_tags:
    if tag["label"].lower() == normalized_label:
        return tag["id"]  # Use existing
# Only create if not found
new_tag = api.create_tag(label=normalized_label)
```

### Pitfall 3: Deleted Tag References
**What goes wrong:** Database stores tag ID, but tag is deleted in Radarr/Sonarr, causing errors when fetching tag label
**Why it happens:** No foreign key constraint between MediaImportSettings.default_tag_id and Radarr/Sonarr tags
**How to avoid:** Handle missing tags gracefully - if tag fetch fails, display placeholder or allow user to re-enter
**Warning signs:** Config page fails to load or shows empty tag field when tag ID exists in database

**Example:**
```python
# BAD - crashes if tag deleted
tag = api.get_tag(id_=import_settings.default_tag_id)
tag_label = tag["label"]  # KeyError if tag is None

# GOOD - handle missing tag
tag_label = None
if import_settings.default_tag_id:
    try:
        tag = api.get_tag(id_=import_settings.default_tag_id)
        tag_label = tag["label"] if tag else None
    except Exception:
        tag_label = None  # Tag deleted or API error
```

### Pitfall 4: Empty String vs None Handling
**What goes wrong:** Saving empty string "" instead of None when user clears tag field, causing database constraint issues
**Why it happens:** JavaScript sends empty string for cleared inputs; backend doesn't convert to None
**How to avoid:** Check for empty/whitespace strings and convert to None before database save
**Warning signs:** Database errors or tag_id column contains empty values

**Example:**
```python
# BAD - stores empty string
tag_label = data.get("tag_label")  # Could be ""
tag_id = create_or_get_tag_id(base_url, api_key, tag_label)  # Error if empty

# GOOD - handle empty string
tag_label = data.get("tag_label")
if tag_label and tag_label.strip():
    tag_id = create_or_get_tag_id(base_url, api_key, tag_label.strip())
else:
    tag_id = None  # User cleared tag field
```

### Complete Create-Or-Get Pattern
```python
# Source: Derived from existing codebase patterns in services/radarr_service.py

from pyarr import RadarrAPI
import logging

logger = logging.getLogger(__name__)

def create_or_get_tag_id(base_url: str, api_key: str, tag_label: str):
    """
    Creates tag if it doesn't exist, or returns existing tag ID.

    Handles:
    - Case-insensitive label matching
    - Label normalization (lowercase, hyphens)
    - Error handling with logging

    Returns:
        int: Tag ID on success
        None: On error (API failure, validation error)
    """
    if not base_url.endswith("/"):
        base_url += "/"

    # Normalize label to API requirements
    normalized_label = tag_label.lower().strip().replace(" ", "-")

    # Remove consecutive hyphens
    while "--" in normalized_label:
        normalized_label = normalized_label.replace("--", "-")

    # Remove leading/trailing hyphens
    normalized_label = normalized_label.strip("-")

    # Validate normalized label
    if not normalized_label:
        logger.warning("Tag label is empty after normalization")
        return None

    try:
        api = RadarrAPI(host_url=base_url, api_key=api_key)

        # Fetch all existing tags
        existing_tags = api.get_tag()

        # Search for existing tag (case-insensitive)
        for tag in existing_tags:
            if tag["label"].lower() == normalized_label.lower():
                logger.info(f"Tag '{normalized_label}' already exists with ID {tag['id']}")
                return tag["id"]

        # Tag doesn't exist - create it
        new_tag = api.create_tag(label=normalized_label)
        logger.info(f"Created new tag '{normalized_label}' with ID {new_tag['id']}")
        return new_tag["id"]

    except Exception as e:
        logger.error(f"Error creating/getting tag '{tag_label}': {e}", exc_info=True)
        return None
```

**Deprecated/outdated:**

## Open Questions

Things that couldn't be fully resolved:

1. **Tag ID Foreign Key Constraint**
   - What we know: `MediaImportSettings.default_tag_id` has foreign key to `tags.id` table
   - What's unclear: Is this a local tag cache table, or does it reference Radarr/Sonarr tag IDs directly?
   - Recommendation: Review `tag_model.py` - it appears to be a local cache with `tag_id` (Radarr/Sonarr ID) and `label`. The foreign key likely references this cache table. If so, the create-or-get pattern should also update the local cache.

2. **Tag Normalization Edge Cases**
   - What we know: Spaces → hyphens, lowercase required
   - What's unclear: How to handle other special characters (periods, underscores, etc.)? Strip them or reject?
   - Recommendation: Strip all non-alphanumeric except hyphens. Test with Radarr/Sonarr API to confirm behavior.

3. **Tag Deletion Synchronization**
   - What we know: Tags can be deleted in Radarr/Sonarr UI
   - What's unclear: Should Listarr detect deleted tags and clear `default_tag_id` automatically?
   - Recommendation: Handle gracefully at fetch time (return None if tag deleted). Consider background sync job in future phases.

4. **Local Tag Cache Usage**
   - What we know: `tag_model.py` exists with service/tag_id/label columns
   - What's unclear: Should the create-or-get pattern update this cache? Is it used elsewhere?
   - Recommendation: Review existing tag cache usage in wizard routes. If cache is used, update it when creating/fetching tags.
