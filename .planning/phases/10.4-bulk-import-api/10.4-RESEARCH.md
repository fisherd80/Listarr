# Phase 10.4: Bulk Import API - Research

**Researched:** 2026-02-08
**Domain:** Radarr/Sonarr API integration, bulk import optimization
**Confidence:** MEDIUM

## Summary

Phase 10.4 aims to replace Listarr's current sequential single-item import approach (one POST per movie/series with 200ms delays) with bulk import API endpoints used by Kometa's arrapi library. Research confirms that both Radarr and Sonarr provide bulk import endpoints (`POST /api/v3/movie/import` and `POST /api/v3/series/import`) that accept arrays of items and return categorized results.

**Current Listarr approach:** Sequential loop calling `add_movie()` or `add_series()` one at a time (200ms delay between calls)
**Kometa approach:** Batch items into groups of up to 100, send single bulk request per batch, parse categorized response

Key findings indicate that while bulk endpoints exist and work, they have practical limitations around batch size (~50-100 items recommended) and resource constraints. The bulk API significantly improves speed by eliminating per-item network overhead and delays, but requires careful batching to avoid timeout/memory issues.

**Primary recommendation:** Implement bulk import with conservative batch size (50 items), maintain existing pre-flight checks as optimization layer, gracefully handle partial failures within bulk responses.

## Architecture Patterns

### Pattern 1: Batch Accumulation with Size Limits

**What:** Accumulate items into batches, send bulk request when batch reaches size limit OR when all items processed

**When to use:** When importing multiple items that can be grouped for efficiency

**Example from Kometa:**
```python
# Kometa's radarr.py pattern (lines 134-145)
movies = []
for i, item in enumerate(tmdb_ids, 1):
    # ... pre-flight checks (skip if exists, on exclusion list, etc.)
    movie = self.api.get_movie(tmdb_id=tmdb_id)
    movies.append(movie)

    # Send batch when reaching limit OR at end of list
    if movies and (len(movies) == 100 or len(tmdb_ids) == i):
        _a, _e, _i, _x = self.api.add_multiple_movies(
            movies, folder, quality_profile, monitor, search,
            availability, tags, per_request=100
        )
        added.extend(_a)
        exists.extend(_e)
        invalid.extend(_i)
        excluded.extend(_x)
        movies = []  # Reset batch
```

**Listarr adaptation:** Same pattern, but use conservative batch size (50) due to timeout concerns

### Pattern 2: Pre-flight Optimization Layer

**What:** Fetch existing IDs and exclusion lists ONCE before import loop, skip known duplicates client-side before bulk call

**When to use:** Always — reduces API load and improves bulk response parsing

**Example from current Listarr:**
```python
# listarr/services/import_service.py _import_movies (lines 262-268)
existing_ids = radarr_service.get_existing_movie_tmdb_ids(base_url, api_key)
excluded_ids = radarr_service.get_exclusions(base_url, api_key)

for item in tmdb_items:
    if tmdb_id in existing_ids:
        result.skipped.append({"tmdb_id": tmdb_id, "title": title, "reason": "already_exists"})
        continue
```

**Maintain this pattern** — bulk endpoint will also skip duplicates, but client-side check eliminates unnecessary lookups and network traffic

### Pattern 3: Lookup Before Bulk Import

**What:** Call `lookup_movie()` or `lookup_series()` for each item BEFORE adding to batch (not after bulk response)

**When to use:** Always — bulk import endpoint requires full item data (not just IDs)

**Current Listarr pattern (lines 309-322):**
```python
# Lookup movie in Radarr (gets full metadata for add payload)
movie_data = radarr_service.lookup_movie(base_url, api_key, tmdb_id)
if not movie_data:
    result.failed.append({"tmdb_id": tmdb_id, "title": title, "reason": "not_found_in_radarr"})
    continue

# Then add movie with full payload
radarr_service.add_movie(base_url, api_key, movie_data, ...)
```

**Bulk adaptation:** Same lookup step, but collect `movie_data` objects into batch array, then send batch to bulk endpoint

### Pattern 4: Categorized Response Parsing

**What:** Bulk endpoint returns items categorized by outcome (added/exists/invalid/excluded), parse each category

**When to use:** After bulk import call completes

**Expected response structure (from arrapi pattern):**
```python
# Bulk import returns 4-tuple
added, exists, invalid, excluded = bulk_import_response

# Map to Listarr's ImportResult structure
for item in added:
    result.added.append({"tmdb_id": item.tmdbId, "title": item.title})
for item in exists:
    result.skipped.append({"tmdb_id": item.tmdbId, "title": item.title, "reason": "already_exists"})
for item in invalid:
    result.failed.append({"tmdb_id": item, "title": "Unknown", "reason": "invalid_id"})
for item in excluded:
    result.skipped.append({"tmdb_id": item, "title": "Unknown", "reason": "on_exclusion_list"})
```

### Anti-Patterns to Avoid

- **Large batches (>100 items):** Radarr/Sonarr have memory/timeout issues with large bulk imports ([GitHub Issue #8885](https://github.com/Radarr/Radarr/issues/8885))
- **No timeout/stop_event checking:** Bulk import can take several minutes — check `stop_event` between batches
- **No activity tracker updates:** Update `activity_tracker` after each batch completes, not just after entire import
- **Removing 200ms delay entirely:** Some delay between batches is still prudent for API health (keep delay between batches, not items)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON array serialization for bulk payload | Custom list->JSON converter | `json=payload_list` in requests.post() | Built-in requests JSON serialization handles it |
| Retry logic for bulk imports | Custom retry loop | Existing http_session retry strategy | Already configured in http_client.py |
| Batch size tuning algorithm | Dynamic batch size adjustment | Fixed batch size (50) | Complexity not justified — fixed size works reliably |
| Response parsing library | Custom JSON parser | Standard dict access + type checking | Response structure is simple list of objects |

**Key insight:** The bulk API is straightforward — it accepts an array of the same payloads used for single-item adds, returns categorized arrays of results. No special libraries needed beyond existing requests/JSON tools.

## Common Pitfalls

### Pitfall 1: Assuming Bulk = Faster Always

**What goes wrong:** Large batches (100+) can timeout or cause Radarr/Sonarr to crash, negating speed gains

**Why it happens:** Radarr/Sonarr bulk import has poor memory management under load ([GitHub Issue #8885](https://github.com/Radarr/Radarr/issues/8885))

**How to avoid:** Use conservative batch size (50 items), allow stop_event checking between batches

**Warning signs:**
- Bulk import requests taking >2 minutes
- HTTP 500 errors from Radarr/Sonarr
- Logs showing connection reset errors

### Pitfall 2: Not Understanding Lookup vs Import

**What goes wrong:** Assuming bulk endpoint accepts TMDB/TVDB IDs directly, getting validation errors

**Why it happens:** Bulk import endpoint requires full item payload (same as single add), not just IDs

**How to avoid:** Call `lookup_movie()` or `lookup_series()` for each ID BEFORE batching, collect full `movie_data` objects

**Warning signs:** API errors like "missing required field: titleSlug" or "invalid payload"

### Pitfall 3: Losing Per-Item Error Context

**What goes wrong:** Bulk response says "3 items failed" but doesn't specify which ones or why

**Why it happens:** Bulk endpoint categorizes by outcome, but may not return detailed per-item errors

**How to avoid:**
- Log each item added to batch with its position/ID
- Parse categorized response and match items back to original list
- Maintain fallback: if bulk fails entirely, could retry failed items individually

**Warning signs:** Job items showing "failed" with no reason, users unable to identify which specific items failed

### Pitfall 4: Forgetting Series TVDB Translation

**What goes wrong:** Batching TMDB IDs for series import, but bulk endpoint expects TVDB IDs

**Why it happens:** Sonarr uses TVDB, not TMDB — translation must happen before lookup

**How to avoid:**
- Keep existing TMDB→TVDB translation step in `_import_series()`
- Lookup series by TVDB ID, not TMDB ID
- Track both IDs in result for logging

**Warning signs:** All series imports failing with "not found" errors

### Batch Loop with Stop Event

```python
# Adaptation from Kometa pattern with Listarr's stop_event checking
BATCH_SIZE = 50
batch = []
batch_start_idx = 0

for idx, item in enumerate(tmdb_items):
    # Check for timeout/cancellation BEFORE processing item
    if stop_event and stop_event.is_set():
        logger.info("Import stopped: timeout or cancellation signal received")
        break

    # Pre-flight checks (skip if exists, excluded, etc.)
    if tmdb_id in existing_ids or tmdb_id in excluded_ids:
        result.skipped.append(...)
        continue

    # Lookup full item data
    movie_data = radarr_service.lookup_movie(base_url, api_key, tmdb_id)
    if not movie_data:
        result.failed.append(...)
        continue

    # Add to batch
    batch.append({
        "title": movie_data.get("title"),
        "tmdbId": movie_data.get("tmdbId"),
        # ... rest of payload fields
    })

    # Send batch when full OR at end of list
    if len(batch) >= BATCH_SIZE or idx == len(tmdb_items) - 1:
        # Import batch
        bulk_result = radarr_service.bulk_add_movies(
            base_url, api_key, batch, ...
        )

        # Parse results
        result.added.extend(bulk_result["added"])
        result.skipped.extend(bulk_result["exists"])
        result.failed.extend(bulk_result["invalid"])

        # Update activity tracker for all items in batch
        if activity_tracker:
            activity_tracker.update(count=len(batch))

        # Reset batch
        batch = []

        # Brief delay between batches (not between items)
        time.sleep(0.5)  # 500ms between batches
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Sequential single-item POST with 200ms delays | Bulk POST with batched payloads | Kometa adopted ~2023 | 10-20x speed improvement for large imports |
| No categorized response parsing | Parse added/exists/invalid/excluded from bulk response | Native to bulk endpoints | Better error reporting granularity |
| Per-item network overhead (~300ms per movie) | Single network call per 50-100 items (~3-5 seconds per batch) | Bulk API inherent benefit | Reduced network latency impact |

**Current industry standard:** Bulk import with conservative batch sizes (50-100 items) and graceful degradation for failures

**Deprecated/outdated:**
- **Sequential one-at-a-time imports:** Still works, but inefficient for lists >20 items
- **200ms artificial delay per item:** Necessary with single-item approach, removed with bulk (delay between batches instead)
- **Assuming unlimited batch size:** Early bulk implementations tried 500+ items, causing crashes — now industry standard is 50-100

## Open Questions

### 1. **Sonarr Bulk Import Endpoint Confirmation**

**What we know:**
- Radarr has confirmed `POST /api/v3/movie/import` endpoint
- Kometa's arrapi has `add_multiple_series()` method suggesting Sonarr equivalent exists
- Sonarr GitHub issues mention lack of ImportList series management endpoints ([Issue #7230](https://github.com/Sonarr/Sonarr/issues/7230))

**What's unclear:**
- Is `POST /api/v3/series/import` the actual endpoint, or different path?
- Does Sonarr bulk import response format match Radarr's categorized structure?

**Recommendation:**
- Test Sonarr bulk import during implementation
- If endpoint doesn't exist or differs significantly, keep sequential approach for Sonarr
- Document findings in implementation plan

**Confidence:** LOW — Sonarr bulk endpoint less documented than Radarr

### 2. **Exact Bulk Response JSON Structure**

**What we know:**
- Kometa parses response into 4 categories (added, exists, invalid, excluded)
- Response is list of objects, not simple ID arrays
- Each returned object has at least `tmdbId`/`tvdbId` and `title` fields

**What's unclear:**
- Complete JSON schema for bulk response
- Whether HTTP 207 Multi-Status or 200 OK with mixed results
- Error detail level for individual failed items

**Recommendation:**
- Test bulk endpoint with small batch (5-10 items) to observe response
- Log full response JSON during testing
- Build response parser based on observed structure

**Confidence:** MEDIUM — Pattern clear from arrapi usage, exact schema needs verification

### 3. **Optimal Batch Size for Listarr's Use Case**

**What we know:**
- Kometa uses 100 items per batch
- GitHub issues report crashes with 100+ items
- Resource constraints (CPU/RAM) affect practical limit

**What's unclear:**
- Whether 50 or 100 is safer default for typical homelab Radarr/Sonarr instances
- Whether batch size should be configurable (environment variable)
- Impact of searchOnAdd=true on batch processing time

**Recommendation:**
- Start with conservative 50 items per batch
- Consider making batch size configurable via environment variable (IMPORT_BATCH_SIZE)
- Test with searchOnAdd=true vs false to measure impact

**Confidence:** MEDIUM — 50-100 range safe based on community reports, exact optimum varies by setup

### 4. **Handling Partial Batch Failures**

**What we know:**
- Bulk endpoint categorizes results, so partial success is expected
- Individual items can fail while others succeed in same batch

**What's unclear:**
- Whether failed items should be retried individually as fallback
- How to preserve per-item error context from bulk response
- Whether to continue to next batch if current batch has high failure rate

**Recommendation:**
- Log all batch results with item-level detail
- DO NOT retry failed items individually (accept bulk response as final)
- Continue processing remaining batches even if current batch has failures
- Surface item-level failure reasons in Job results

**Confidence:** HIGH — This is standard bulk API pattern, no retry needed
