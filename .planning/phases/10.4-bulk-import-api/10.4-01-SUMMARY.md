# Phase 10.4 Plan 01: Bulk Import API Implementation Summary

**One-liner:** Replace sequential single-item POSTs with batch-based bulk import endpoints (50 items/batch) for dramatic import speed improvement

## What Was Built

Implemented bulk import functionality for Radarr and Sonarr, replacing the sequential single-item POST pattern with batch-based bulk import endpoints that send up to 50 items per API call.

### Components Added

**Service Layer Functions:**
- `bulk_add_movies()` in radarr_service.py - POSTs to `/api/v3/movie/import` with movie payload array
- `bulk_add_series()` in sonarr_service.py - POSTs to `/api/v3/series/import` with series payload array
- `_flush_movie_batch()` helper in import_service.py - Processes bulk movie import results
- `_flush_series_batch()` helper in import_service.py - Processes bulk series import results

**Constants:**
- `BULK_TIMEOUT = 300` in http_client.py - 5-minute timeout for bulk operations
- `BATCH_SIZE = 50` in import_service.py - Items per batch (conservative vs Kometa's 100)
- `BATCH_DELAY = 0.5` in import_service.py - 500ms delay between batches

**Import Pipeline Refactor:**
- Rewrote `_import_movies()` to use batch accumulation pattern
- Rewrote `_import_series()` to use batch accumulation pattern
- Pre-flight checks (existing_ids, excluded_ids) still operate per-item before lookup
- Lookup operations still happen individually (TMDB-to-TVDB translation required for Sonarr)
- Payloads accumulated during lookup phase, flushed in batches of 50
- stop_event checked per-iteration, pending batch flushed before break

### Behavior Changes

**Before (Sequential):**
- Lookup → Add → 200ms delay → Lookup → Add → 200ms delay (per item)
- One POST request per item
- HTTPError handling for MovieExistsValidator/SeriesExistsValidator

**After (Batch):**
- Lookup → Accumulate → Lookup → Accumulate (batch of 50)
- Flush batch → Bulk POST (50 items) → 500ms delay
- Duplicate detection via response parsing (items not in response = already exists)
- No per-item add delay, only per-batch delay

**Expected Performance:**
- 100 items: ~100 seconds → ~10 seconds (10x faster)
- Large lists benefit most (fewer API calls, less delay overhead)

### Batch Accumulation Pattern

```python
# Phase 1: Pre-flight
existing_ids = service.get_existing_movie_tmdb_ids(...)
excluded_ids = service.get_exclusions(...)

# Phase 2: Lookup + Accumulate
batch = []
batch_meta = []
for item in tmdb_items:
    # Check stop_event - flush pending batch if stopping
    # Check pre-flight filters (existing/excluded) - skip if matched
    # Lookup in Radarr/Sonarr
    # Build payload dict (9 fields for movies, 11 for series)
    batch.append(payload)
    batch_meta.append({tmdb_id, title})

    # Flush if batch full
    if len(batch) >= BATCH_SIZE:
        _flush_batch(...)
        batch = []
        batch_meta = []
        time.sleep(BATCH_DELAY)

# Phase 3: Flush remainder
if batch:
    _flush_batch(...)
```

### Result Processing

Bulk endpoints return only successfully added items. Items not in the response are treated as skipped (already_exists):

```python
added_items = bulk_add_movies(base_url, api_key, batch)
added_tmdb_ids = {item.get("tmdbId") for item in added_items}

for meta in batch_meta:
    if meta["tmdb_id"] in added_tmdb_ids:
        result.added.append(...)
    else:
        result.skipped.append({..., "reason": "already_exists"})
```

### Error Handling

- Bulk operation failure → All items in batch marked as failed
- Lookup failure → Item marked as failed, not included in batch
- Pre-flight filter match → Item marked as skipped, not looked up

## Decisions Made

### Batch Size Selection

**Choice:** 50 items per batch

**Rationale:**
- Conservative limit vs Kometa's 100
- Balances throughput with timeout risk
- 50 items × ~2-3 seconds per item = ~120 seconds worst case (within 300s timeout)
- Server-side processing time varies with search_on_add setting

**Alternatives Considered:**
- 25 items: Too conservative, minimal performance gain
- 100 items: Matches Kometa but higher timeout risk with slow servers

### Timeout Configuration

**Choice:** 300 seconds (5 minutes) for bulk operations

**Rationale:**
- Bulk endpoint processes entire batch server-side
- Each item may trigger search if search_on_add=true
- 50 items × 3 seconds = 150 seconds baseline + buffer
- ADD_TIMEOUT (120s) insufficient for bulk operations

**Alternatives Considered:**
- 120s: Too short for 50-item batches with search enabled
- 600s: Unnecessarily long, masks potential issues

### Duplicate Handling

**Choice:** Remove MovieExistsValidator/SeriesExistsValidator error parsing

**Rationale:**
- Bulk endpoints silently skip duplicates (items not in response = already exists)
- Pre-flight checks already filter most duplicates
- Simpler result processing (response presence = added, absence = skipped)

**Alternatives Considered:**
- Keep error handling: Unnecessary complexity, bulk endpoint doesn't return errors for duplicates

## Key Learnings

1. **Bulk API behavior:** Radarr/Sonarr bulk import endpoints silently skip duplicates rather than returning errors
2. **Timeout scaling:** Bulk operations need significantly longer timeouts than single-item adds
3. **Batch size tradeoffs:** Smaller batches (50 vs 100) reduce timeout risk but increase total execution time
4. **Pre-flight value:** Duplicate checking before lookup prevents wasted API calls

## Performance Expectations

**Sequential (Before):**
- 100 items × (lookup 200ms + add 2s + delay 200ms) = ~240 seconds
- 100 API calls for adds

**Batch (After):**
- 100 items × (lookup 200ms) = 20 seconds
- 2 batches × (bulk add ~5s + delay 500ms) = ~11 seconds
- Total: ~31 seconds
- 2 API calls for adds

**Improvement:** ~8x faster for 100 items (240s → 31s)
