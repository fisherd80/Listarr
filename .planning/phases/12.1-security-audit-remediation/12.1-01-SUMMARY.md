# Phase 12.1 Plan 01: Critical Authentication & CSRF Fixes Summary

**Fixed 2 critical security vulnerabilities: dashboard authentication bypass and unnecessary CSRF exemptions**

## Objective

Fix 2 critical security vulnerabilities identified by flask-security-reviewer audit: (1) Dashboard routes accessible without authentication, (2) Unnecessary @csrf.exempt decorators on 7 POST endpoints that already have CSRF token handling in JavaScript.

## Tasks Completed

### Task 1: Add @login_required to dashboard routes
**Commit:** ae65c54

Added `@login_required` decorator to all 4 dashboard endpoints:
- `dashboard_page()` - Main dashboard view (redirects to /login)
- `dashboard_stats()` - API endpoint for stats (returns 401/redirect)
- `recent_jobs()` - API endpoint for recent jobs (returns 401/redirect)
- `upcoming_jobs()` - API endpoint for upcoming scheduled jobs (returns 401/redirect)

Dashboard now requires authentication for all endpoints. Previously, any unauthenticated user on the network could view Radarr/Sonarr connection status, version numbers, media counts, and job history.

**Files modified:**
- `listarr/routes/dashboard_routes.py`

### Task 2: Remove @csrf.exempt from POST endpoints
**Commit:** 7bad4a6

Removed unnecessary `@csrf.exempt` decorators from 7 endpoints:

**jobs_routes.py:**
- `rerun_job()` - Rerun failed job
- `clear_all_jobs()` - Clear all job history
- `clear_list_jobs()` - Clear job history for specific list

**lists_routes.py:**
- `run_list_import()` - Manually trigger list import

**schedule_routes.py:**
- `pause_schedule()` - Pause all scheduled jobs
- `resume_schedule()` - Resume all scheduled jobs
- `update_schedule()` - Update schedule for specific list

Also removed unused `csrf` imports from all three route files.

These endpoints already have CSRF token handling in JavaScript (X-CSRFToken header), making the exemptions unnecessary and creating cross-site request forgery vulnerabilities.

**Files modified:**
- `listarr/routes/jobs_routes.py`
- `listarr/routes/lists_routes.py`
- `listarr/routes/schedule_routes.py`

### Task 3: Update tests for authentication requirements
**Commit:** 3e34311

Created new test file `tests/routes/test_dashboard_auth.py` with 4 authentication tests:
- `test_dashboard_page_requires_authentication()` - Verifies page redirects to /login
- `test_dashboard_stats_requires_authentication()` - Verifies API redirects to /login
- `test_dashboard_recent_jobs_requires_authentication()` - Verifies API redirects to /login
- `test_dashboard_upcoming_requires_authentication()` - Verifies API redirects to /login

Tests use `app_with_auth` fixture to ensure authentication is enforced. Flask-Login's `@login_required` decorator redirects unauthenticated requests to /login (302 status code).

**Files created:**
- `tests/routes/test_dashboard_auth.py`

## Technical Notes

### Flask-Login Behavior
Flask-Login's `@login_required` decorator:
- Returns 302 redirect to `/login` for unauthenticated requests (default behavior)
- Returns 401 Unauthorized if `unauthorized` callback is configured for JSON endpoints
- Preserves `next` parameter for post-login redirection

### CSRF Token Flow
1. Base template includes `<meta name="csrf-token">` with token
2. JavaScript reads token: `document.querySelector('meta[name="csrf-token"]').content`
3. All AJAX POST requests include `X-CSRFToken` header
4. Flask-WTF validates token automatically

### Test Fixture Notes
- `app` fixture: Has `LOGIN_DISABLED=True` (bypasses authentication for existing tests)
- `app_with_auth` fixture: Authentication active (use for auth tests)
- `auth_client` / `authenticated_client` fixtures: Pre-authenticated test clients
