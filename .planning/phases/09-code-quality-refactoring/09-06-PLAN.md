```python
def _test_and_update_service_status(service: str, base_url: str, api_key: str):
    """
    Test API connection and update database status for a service.

    Args:
        service: Service identifier ("RADARR" or "SONARR")
        base_url: Service base URL
        api_key: Service API key
    """
    from listarr.services.radarr_service import validate_radarr_api_key
    from listarr.services.sonarr_service import validate_sonarr_api_key

    # Select the appropriate validation function
    validate_fn = validate_radarr_api_key if service == "RADARR" else validate_sonarr_api_key

    try:
        test_result = validate_fn(base_url, api_key)

        # Update database status
        service_config = ServiceConfig.query.filter_by(service=service).first()
        if service_config:
            service_config.api_test_success = test_result
            service_config.api_test_last_run = datetime.now(timezone.utc)
            db.session.commit()
            current_app.logger.info(f"{service} API test {'passed' if test_result else 'failed'}")
    except Exception as e:
        current_app.logger.error(f"Error testing {service} API: {e}", exc_info=True)
        # Update status to failed
        service_config = ServiceConfig.query.filter_by(service=service).first()
        if service_config:
            service_config.api_test_success = False
            service_config.api_test_last_run = datetime.now(timezone.utc)
            db.session.commit()
```

2. Keep the old function names as thin wrappers for backwards compatibility:

```python

    """Test Radarr API and update database status."""

def _test_and_update_sonarr_status(base_url: str, api_key: str):
    """Test Sonarr API and update database status."""
    _test_and_update_service_status("SONARR", base_url, api_key)
```

NOTE: Since SQLite doesn't support dropping columns directly, and the columns are unused, the simplest approach is:
- Remove from the model (Python code)
- The columns will remain in SQLite but won't be accessed
- For a clean database, users can delete listarr.db and it will be recreated without these columns

This is acceptable because:
1. The columns have never been used (no data loss)
2. SQLite column removal requires table rebuild
3. Existing databases will continue to work (extra columns are ignored)
