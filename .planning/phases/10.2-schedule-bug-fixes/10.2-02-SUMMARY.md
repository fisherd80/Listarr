# Phase 10.2 Plan 02: Activity-Based Idle Timeout for Jobs Summary

**One-liner:** Replace fixed 10-minute wall-clock timer with activity-based idle detection (5-min idle) and add stop_event awareness to import loops for early termination

## Objective

Replace the static 10-minute wall-clock timeout with an activity-based idle timeout that only triggers when the job makes no progress for a configurable idle period, and add stop_event awareness to import loops for early termination.

**Problem:** The current fixed timer incorrectly marks large but actively-progressing jobs as "failed/timed out" because the timer fires unconditionally after 10 minutes. A 500-item import taking 15 minutes completes all items successfully but gets labeled as failed.

**Solution:** Track last activity time and only timeout idle/hung jobs. Import loops check stop_event and update activity_tracker after each item.

## Implementation Summary

### Task 1: Activity-Based Idle Timeout in job_executor.py

**Changes:**
- Replaced `JOB_TIMEOUT_SECONDS = 600` with `IDLE_TIMEOUT_SECONDS = 300` and `IDLE_CHECK_INTERVAL = 30`
- Added `ActivityTracker` class with thread-safe `update()` and `idle_seconds` property
- Replaced `_trigger_timeout()` with `_monitor_idle()` function that checks `activity_tracker.idle_seconds`
- Updated `submit_job()` to create `ActivityTracker` and spawn monitor thread (not fixed Timer)
- Updated `_execute_job()` signature to accept `activity_tracker` and pass to `import_list()`
- Updated `_mark_job_timeout()` error message to reference idle timeout

**Design:**
- `_monitor_idle()` uses `monitor_stop.wait(IDLE_CHECK_INTERVAL)` instead of `time.sleep()` for immediate cleanup
- Monitor thread exits via `monitor_stop.set()` in cleanup callback
- `IDLE_TIMEOUT_SECONDS = 300` (5 min idle) catches hung jobs without false positives
- Thread-safe `ActivityTracker` uses lock (tracker updated from executor thread, read from monitor thread)

### Task 2: stop_event and Activity Tracking in import_service.py

**Changes:**
- Updated `import_list()`, `_import_movies()`, `_import_series()` to accept `stop_event=None, activity_tracker=None`
- Added `stop_event.is_set()` check at top of both import loops (break early on timeout)
- Added `activity_tracker.update()` after each item outcome:
  - After `result.failed.append()` (no tmdb_id, lookup failed)
  - After `result.skipped.append()` (already_exists)
  - After try/except block (added or exception)
- Backward compatible with None defaults (no crash if parameters not provided)

**Impact:**
- Import loops terminate early when idle timeout triggers
- Activity tracker updated for ALL outcomes (added, skipped, failed) — any progress is activity
- Partial results preserved in ImportResult before break

## Behavioral Changes

**Before:**
- All jobs marked "failed/timed out" after exactly 10 minutes (600s)
- Long-running imports (500+ items, 15+ minutes) incorrectly labeled as failed despite completing successfully
- No way to interrupt import loops early

**After:**
- Jobs only timeout after 5 minutes (300s) of NO ACTIVITY
- Large imports (500+ items) complete successfully as "completed" status
- Stalled/hung jobs correctly detected and timed out
- Import loops check stop_event and break early when timeout triggers
- Activity tracked after every item processed (added, skipped, or failed)

## Lessons Learned

1. **Activity-based timeouts are more robust than wall-clock timers** - They distinguish between "taking a long time" and "actually hung"
2. **Thread-safe tracking requires proper locking** - ActivityTracker._lock ensures safe concurrent access
3. **Event.wait() vs time.sleep()** - Using Event.wait(interval) allows immediate cleanup when monitor_stop is set
4. **Update activity on ALL outcomes** - Any progress (added/skipped/failed) is activity, prevents false positives
5. **Backward compatibility with None defaults** - Allows gradual rollout and testing
