## Decisions Made

**1. Use explicit 9-field payload for add_movie**
- Rationale: Copying ALL fields from lookup (including 'id') causes 400 Bad Request
- Implementation: Explicit mapping of title, tmdbId, year, qualityProfileId, titleSlug, images, rootFolderPath, monitored, addOptions, tags
- Impact: Fixes Radarr 400 errors that prevented all movie imports

**2. Use explicit 11-field payload for add_series**
- Rationale: Sonarr requires 'seasons' array and uses 'tvdbId' (not tmdbId)
- Implementation: Explicit mapping including tvdbId, seasons, seasonFolder
- Impact: Prevents future Sonarr payload issues similar to Radarr bug

**3. Log error response body before raise_for_status**
- Rationale: HTTP error messages from Radarr/Sonarr API contain debugging details
- Implementation: Check response.ok, log response.text before raise_for_status
- Impact: Debugging aid for future API errors

**4. Check exclusion list AFTER existing items check**
- Rationale: Existing items check is in-memory (fast), exclusion list requires API call
- Implementation: Placed exclusion check after existing_ids check in import loop
- Impact: Minor optimization (no API call wasted on already-existing items)

**5. Return empty set from get_exclusions on error**
- Rationale: Caller expects set type, empty set is safe default (import proceeds)
- Implementation: Return set() instead of None when API call fails
- Impact: Import continues even if exclusion list fetch fails

**6. Use 'on_exclusion_list' as skip reason**
- Rationale: Distinct from 'already_exists' for job tracking and debugging
- Implementation: Separate skip reason in ImportResult.skipped entries
- Impact: Clear distinction in job results between duplicates and blocked content
