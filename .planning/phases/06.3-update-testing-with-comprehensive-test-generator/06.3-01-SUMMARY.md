## Technical Decisions

### Decision 1: Coverage Baseline Before Test Generation
**Rationale:** Running coverage analysis BEFORE writing new tests prevents:
- Writing tests for already-covered code
- Missing critical uncovered paths
- Inefficient test generation efforts

**Result:** Clear map of exactly which lines need coverage

### Decision 2: Prioritize by Impact and Dependency
**Rationale:** Test generation order matters:
- TMDB service functions (Plan 02) are called by cache layer
- TMDB cache functions (Plan 03) are called by import service
- Import service is called by wizard routes
- Bottom-up approach ensures foundation is solid

**Result:** 4 test generation groups in dependency order

### Decision 3: Target 90%+ Coverage for Phase 6.2 Critical Paths
**Rationale:** Phase 6.2 features are customer-facing and recently added:
- Top rated presets are new list types
- Region filtering affects data quality
- Limit options affect user experience
- Higher coverage reduces regression risk

**Result:** Set 90%+ target for Phase 6.2 modules, 85%+ overall
