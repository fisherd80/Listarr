**Researched:** 2026-02-01
**Domain:** Python testing with pytest, test coverage analysis, test gap identification

## Summary

This phase focuses on expanding test coverage for Listarr using the project's existing comprehensive-test-generator agent to ensure robust testing across all features implemented in Phases 1-6.2. The research reveals that the project already has a strong testing foundation with 415 passing tests (~8,617 lines of test code) using pytest, pytest-cov, and established testing patterns. The primary challenge is identifying coverage gaps and systematically expanding tests for newer features (Phase 6.2 list enhancements) that currently lack dedicated test coverage.

The standard approach combines pytest-cov's coverage reporting capabilities with the comprehensive-test-generator agent to identify untested code paths, then generate well-structured tests following the project's existing patterns. The project already has excellent testing infrastructure (conftest.py fixtures, mocking patterns, test organization) that should be leveraged when expanding coverage.

**Primary recommendation:** Use pytest-cov with term-missing and HTML reports to identify coverage gaps, prioritize testing for Phase 6.2 features (top_rated presets, region filtering, limit options), then invoke the comprehensive-test-generator agent to create tests following established project patterns.

### Pattern 3: Fixture-Based Testing (Project Standard)
**What:** Use pytest fixtures for test isolation and reusability
**When to use:** All tests should leverage existing fixtures
**Example:**
```python
# Source: tests/conftest.py (existing project pattern)
@pytest.fixture(scope='function')
def app(temp_instance_path):
    test_config = {
        'TESTING': True,
        'SQLALCHEMY_DATABASE_URI': 'sqlite:///:memory:',
        'WTF_CSRF_ENABLED': False,
        'SECRET_KEY': 'test-secret-key'
    }
    app = create_app(test_config=test_config)
    app.instance_path = temp_instance_path

    with app.app_context():
        db.create_all()
        yield app
        db.session.remove()
        db.drop_all()

# Usage in tests
def test_something(app, client):
    # app and client fixtures automatically provide isolated environment
    response = client.get('/endpoint')
    assert response.status_code == 200
```

### Pattern 4: API Mocking (Project Standard)
**What:** Mock external API calls to avoid dependencies and ensure speed
**When to use:** Any test that would call TMDB, Radarr, or Sonarr APIs
**Example:**
```python
# Source: tests/unit/test_tmdb_service.py (existing project pattern)
from unittest.mock import patch, MagicMock

@patch('listarr.services.tmdb_service.Movie')
def test_get_top_rated_movies(mock_movie_class):
    mock_movie = MagicMock()
    mock_movie.top_rated.return_value = [
        {'id': 1, 'title': 'Test Movie', 'vote_average': 9.0}
    ]
    mock_movie_class.return_value = mock_movie

    result = get_top_rated_movies("test_key")

    assert len(result) == 1
    assert result[0]['title'] == 'Test Movie'
    mock_movie.top_rated.assert_called_once()
```

### Pattern 5: Arrange-Act-Assert Structure (Project Standard)
**What:** Clear three-phase test structure for readability
**When to use:** All tests should follow this pattern
**Example:**
```python
# Source: Project standard pattern
def test_cache_hit_prevents_api_call(app, mocker):
    # Arrange
    mock_fetch = mocker.patch('listarr.services.tmdb_cache.get_top_rated_movies')
    mock_fetch.return_value = [{'id': 1, 'title': 'Cached'}]

    # Act
    result1 = get_top_rated_movies_cached("test_key")
    result2 = get_top_rated_movies_cached("test_key")  # Should hit cache

    # Assert
    assert result1 == result2
    mock_fetch.assert_called_once()  # Only one API call despite two requests
```

### Anti-Patterns to Avoid
- **Don't test implementation details**: Test behavior, not internal variables
- **Don't skip mocking external APIs**: Real API calls make tests slow and flaky
- **Don't share state between tests**: Each test must be independent
- **Don't use generic test names**: Use descriptive names like `test_cache_returns_empty_list_when_api_fails`
- **Don't ignore coverage for "simple" code**: Simple code can have edge cases

## Common Pitfalls

### Pitfall 1: Testing Without Coverage Analysis First
**What goes wrong:** Writing tests for already-covered code while missing critical gaps
**Why it happens:** Diving into test creation without systematic gap identification
**How to avoid:** Always run coverage analysis first to identify actual gaps
**Warning signs:** New tests don't improve coverage percentage significantly

### Pitfall 2: Ignoring Branch Coverage
**What goes wrong:** 100% line coverage but untested conditional branches
**Why it happens:** pytest-cov defaults to line coverage, not branch coverage
**How to avoid:** Use `--cov-branch` flag for branch coverage measurement
**Warning signs:** Tests pass but bugs occur in if/else branches

### Pitfall 3: Over-Mocking Integration Tests
**What goes wrong:** Integration tests that mock so much they test nothing
**Why it happens:** Confusion between unit tests (mock everything) and integration tests (mock external only)
**How to avoid:** Integration tests should only mock external services (TMDB, Radarr, Sonarr), not internal components
**Warning signs:** Integration test can't catch bugs in component interactions

### Pitfall 4: Not Following Project Patterns
**What goes wrong:** Tests that don't match existing style, making maintenance harder
**Why it happens:** Not reviewing existing tests before writing new ones
**How to avoid:** Study tests/unit/test_tmdb_service.py and tests/routes/test_config_routes.py for patterns
**Warning signs:** Different fixture usage, mocking style, or test naming than existing tests

### Pitfall 5: Testing Phase 6.2 Features in Wrong Layer
**What goes wrong:** Testing new features only at route level, missing service/cache layer tests
**Why it happens:** Not understanding the application's layered architecture
**How to avoid:** Test each layer independently:
  - Unit: tmdb_service.get_top_rated_movies
  - Unit: tmdb_cache.get_top_rated_movies_cached (with mocked service)
  - Route: lists_routes preview endpoint (with mocked cache)
  - Integration: Full flow from route → cache → service
**Warning signs:** Coverage shows untested lines in service/cache layers

### Pitfall 6: Incomplete Phase 6.2 Test Coverage
**What goes wrong:** Missing tests for specific Phase 6.2 features (region filtering, limit validation)
**Why it happens:** Not reviewing Phase 6.2 PLAN.md files to understand what was added
**How to avoid:** Review .planning/phases/06.2-list-enhancements/06.2-0*-PLAN.md to understand:
  - Top rated presets (movies/TV)
  - Region setting in TMDB calls
  - Limit options (100, 250, 500, 1000)
  - Preset UI changes
**Warning signs:** Coverage reports show tmdb_cache._get_tmdb_region() untested

```python
from unittest.mock import patch, MagicMock
from listarr.services.tmdb_service import get_top_rated_movies

@patch('listarr.services.tmdb_service.Movie')
def test_get_top_rated_movies_returns_results(mock_movie_class):
    """Top rated movies fetches from TMDB top_rated endpoint."""
    # Arrange
    mock_movie = MagicMock()
    mock_movie.top_rated.return_value = [
        {'id': 238, 'title': 'The Godfather', 'vote_average': 8.7},

    mock_movie_class.return_value = mock_movie

    # Act
    result = get_top_rated_movies("test_api_key", page=1)

    # Assert
    assert len(result) == 2
    assert result[0]['title'] == 'The Godfather'
    mock_movie.top_rated.assert_called_once_with(page=1)

@patch('listarr.services.tmdb_service.Movie')
def test_get_top_rated_movies_handles_api_error(mock_movie_class):
    """Top rated movies returns empty list on API error."""
    # Arrange
    mock_movie = MagicMock()
    mock_movie.top_rated.side_effect = Exception("API error")
    mock_movie_class.return_value = mock_movie

    # Act
    result = get_top_rated_movies("test_api_key")

    # Assert
    assert result == []
```

### Test for Cached Top Rated with Region
```python
# Source: Project pattern from tmdb_cache tests (if they exist)
# New integration test for Phase 6.2
from unittest.mock import patch, MagicMock
from listarr.services.tmdb_cache import get_top_rated_movies_cached

@patch('listarr.services.tmdb_cache.get_top_rated_movies')
@patch('listarr.services.tmdb_cache._get_tmdb_region')
def test_cached_top_rated_uses_region_in_cache_key(mock_region, mock_fetch, app):
    """Cache key includes region to prevent cross-region cache contamination."""
    # Arrange
    mock_region.return_value = 'FR'
    mock_fetch.return_value = [{'id': 1, 'title': 'French Top Movie'}]

    with app.app_context():
        # Act - First call with region FR
        result_fr = get_top_rated_movies_cached("test_key", page=1)

        # Change region
        mock_region.return_value = 'US'
        mock_fetch.return_value = [{'id': 2, 'title': 'US Top Movie'}]

        result_us = get_top_rated_movies_cached("test_key", page=1)

        # Assert - Should fetch twice (different cache keys)
        assert result_fr != result_us
        assert mock_fetch.call_count == 2
```
