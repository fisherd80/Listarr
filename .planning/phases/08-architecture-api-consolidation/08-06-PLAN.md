# Phase 8: Architecture Concerns for Phase 9

**Created:** [DATE]
**Context:** Issues identified during API consolidation research
**Action Required:** Evaluate and address in Phase 9 (Code Quality & Refactoring)

## Summary

During Phase 8's API consolidation, several architecture concerns were identified that warrant review in Phase 9. These range from potential over-engineering to inconsistent patterns that affect maintainability.

## Concerns

### 1. Service Layer Abstraction Value

**Observation:** radarr_service.py and sonarr_service.py contain 11-12 thin wrapper functions that mostly translate parameters and call HTTP endpoints.

**Question:** Is this service layer adding value or just indirection? Consider:
- Pro: Testability (mock service, not HTTP)
- Pro: Abstraction of API details
- Con: Extra layer of code to maintain
- Con: Functions are very thin wrappers

**Recommendation:** Keep for now. The testability benefit outweighs the slight overhead. However, consolidate duplicate code between Radarr/Sonarr services (e.g., shared `_make_api_call` helper).

### 2. Duplicate Code: Radarr vs Sonarr

**Observation:** ~326 lines of duplicate code between radarr_service.py and sonarr_service.py (estimated in PROJECT.md).

**Examples:**
- validate_*_api_key() - identical logic, different endpoint
- get_quality_profiles() - identical logic, different endpoint
- get_root_folders() - identical logic, different endpoint
- get_tags() - identical logic, different endpoint
- create_or_get_tag_id() - identical normalization logic

**Recommendation:** Create a base `ArrService` class or shared helper module in Phase 9:
```python
def _arr_api_get(base_url, api_key, endpoint):
    url = f"{normalize_url(base_url)}/api/v3/{endpoint}"
    headers = {"X-Api-Key": api_key}
    response = http_session.get(url, headers=headers, timeout=DEFAULT_TIMEOUT)
    response.raise_for_status()
    return response.json()
```

### 3. Inconsistent Error Handling

**Observation:** Functions return different types on error:
- `get_quality_profiles()` returns `[]` on error
- `get_system_status()` returns `{}` on error
- `lookup_movie()` returns `None` on error
- `add_movie()` raises exception on error

**Impact:** Callers must know which pattern each function uses. Silent failures possible when returning empty collections.

**Recommendation:** Standardize in Phase 9:
- Option A: Always return Result[T, Error] type
- Option B: Always raise exceptions, let callers handle
- Option C: Return None for single-item, [] for collections, raise for mutations

### 4. Module Size: config_routes.py (592 lines)

**Observation:** config_routes.py handles both Radarr and Sonarr configuration in a single file, leading to large module size.

**Recommendation:** Split in Phase 9:
- `radarr_config_routes.py` - Radarr configuration
- `sonarr_config_routes.py` - Sonarr configuration
- Or: Extract shared logic into helpers

### 5. Dashboard Cache: Synchronous Startup

**Observation:** `initialize_dashboard_cache()` blocks app startup while fetching stats from Radarr/Sonarr.

**Impact:** If services are slow/down, Flask app startup hangs. Poor user experience in Docker when services aren't ready.

**Recommendation:** Move to background thread or defer until first dashboard load:
```python
# Option A: Background refresh
threading.Thread(target=initialize_dashboard_cache, daemon=True).start()

# Option B: Lazy load
def get_dashboard_stats():
    if not _cache_initialized:
        initialize_dashboard_cache()
    return _cached_stats
```

### 6. API Delay: Fixed 200ms Sleep

**Observation:** `API_CALL_DELAY = 0.2` with `time.sleep()` after each TMDB call.

**Impact:** Batch imports of 100 items = 20 seconds of just waiting. May be too conservative (wasting time) or too aggressive (still triggers rate limits).

**Recommendation:** Implement adaptive rate limiting in Phase 9:
- Track 429 responses
- Increase delay when rate limited
- Decrease delay when successful
- Or: Use TMDB's rate limit headers if available

## Priority for Phase 9

| Priority | Concern | Impact | Effort |
|----------|---------|--------|--------|
| HIGH | Duplicate code consolidation | Maintainability | Medium |
| HIGH | Error handling standardization | Reliability | Medium |
| MEDIUM | config_routes.py split | Maintainability | Low |
| LOW | Dashboard cache async | Startup UX | Low |
| LOW | Adaptive rate limiting | Performance | Medium |

## Files Affected

- `listarr/services/dashboard_cache.py`
- `listarr/services/tmdb_cache.py`
